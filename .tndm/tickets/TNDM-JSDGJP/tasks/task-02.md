# Implementation details

## `ids.ts`

Define documented branded string aliases or nominal interfaces for future handles:

- `TargetId`
- `SpanId`
- `GraphNodeId`
- `PlanId`
- `ChangeId`

Document stability expectations:

- IDs are session/workspace-scoped unless a future phase explicitly persists them.
- IDs should be generated by resolution/planning services, not manually constructed by callers.
- IDs are safer than repeated file/line/character coordinates, but must be invalidated on file fingerprint changes where relevant.

## `results.ts`

Define shared structured result skeleton types with JSDoc:

- provenance source: `"semantic" | "structural" | "search" | "git" | "diagnostics" | "heuristic"`
- source range/file span shape
- workflow item shape with file/range/symbol/snippet/reason/provenance
- next-action shape with tool name, reason, and optional params summary
- result envelope shape with markdown, confidence, provenance, items, omitted count, and next actions

Keep these as type-level contracts; no runtime behavior is required.

## `schemas.ts`

Create TypeBox schemas for the planned V2 tools using flat property shapes:

- `CodeResolveParameters`
- `CodeContextParameters`
- `CodeFindParameters`
- `CodeGraphParameters`
- `CodeImpactParameters`
- `CodeRefactorParameters`
- `CodeApplyParameters`
- `CodeHealthParameters`

Use `StringEnum` from `@earendil-works/pi-ai` for string enums, matching existing package conventions. Include JSDoc comments describing cross-field runtime rules that TypeBox does not enforce.

Schema notes:

- use `scope`, not `path`, in V2 skeleton docs
- `code_find.mode` excludes `natural` in Phase 0
- `code_graph.relations` uses `references`, not `callers`
- `code_refactor.operation` is the only `operation` enum; no broad `action` parameter exists
- `code_apply` requires `planId`

## `surface.ts`

Define the canonical planned surface metadata:

- `WORKFLOW_CODE_TOOL_NAMES`
- `WorkflowCodeToolName`
- `WORKFLOW_CODE_TOOL_SPECS`

Each spec should include:

- `name`
- `purpose`
- `schemaKey`
- `absorbs` current tools/behaviors
- `substrates` needed later (`semantic`, `structural`, `search`, `git`, `diagnostics` as applicable)
- `phase`
- `nonGoals`
- concise schema documentation for agents/maintainers

Document that this is a planned skeleton and not the registered public tool surface in Phase 0.

## `index.ts`

Export only the workflow skeleton contracts from `src/workflow/` for tests and future implementation.
