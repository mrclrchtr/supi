# Split CodeSymbol into name and declaration anchors

**Context.** `CodeSymbol` (`@mrclrchtr/supi-code-runtime`) carried a single `{ line, character }` populated inconsistently by the LSP provider: `flattenDocumentSymbols` (in `@mrclrchtr/supi-lsp`) used `DocumentSymbol.selectionRange.start` — the identifier — while `toCodeSymbol` (workspace-symbol hits via `SymbolInformation`) used `location.range.start` — the declaration, including `export`/modifier keywords. Position-strict substrates (tree-sitter `calleesAt`, LSP `rename`) silently produced empty/`unavailable` results when fed a declaration anchor. A temporary instrumented probe in `resolveSymbolTarget` (`packages/supi-code-intelligence/src/targeting/resolve-symbol.ts`) confirmed the precise root cause: `refineResolvedSymbolAnchor` was only invoked on the single-match path (`ranged.length === 1`); the **disambiguation path** (`ranged.length !== 1`) never refined candidates, so they inherited the declaration anchor directly from `toCodeSymbol.location.range.start`, carried unrefined through `registerCandidate`. The single-match path was healthy (probe showed `{78,1}` → `{78,23}` for `buildArchitectureModel`). The unit tests mocked the provider and asserted `displayCharacter` values they fed in, so the class of regression passed CI.

**Decision.** Split the contract: `CodeSymbol` gains `declarationAnchor` (always present) and optional `nameAnchor` (best-effort). Population is layered: the provider populates `declarationAnchor` always and `nameAnchor` when it has `selectionRange` (the `DocumentSymbol` path); for `SymbolInformation` (no `selectionRange`) `nameAnchor` is left undefined and the **orchestration layer** (`supi-code-intelligence/src/targeting`) backfills it via documentSymbols-refine, then a tree-sitter `nodeAt` identifier-snap fallback (second-tier, not mandatory — documentSymbols-refine already works for the single-match case per the probe). The `nodeAt` snap lives in the orchestration layer, not the semantic provider, to keep `SemanticProvider` and `StructuralProvider` decoupled. Both the single-match refine path **and** the disambiguation path refine candidate anchors. Position-strict consumers (`calleesAt`, `rename`) prefer `nameAnchor` and hard-fail with an observable note when absent; position-tolerant consumers (`references`, `implementation`, `definition` — and `code_impact`, which is reference-backed) accept either. `targetId` (`src/workflow/target-store.ts`) hashes identity-stable fields only (`cwd, file, name, kind, container, fingerprint`), demoting position to a display attribute so re-resolution reuses IDs regardless of refine success; `container` is added to the hash (currently absent, risking same-file container collisions). Migration is big-bang (no deprecated `line`/`character` aliases — aliases would re-introduce the conflation). Vocabulary pinned in `CONTEXT.md`: `Name anchor`, `Declaration anchor`.

**Considered Options (rejected).**

- *B — provider-only standardization* (`flattenDocumentSymbols` + `toCodeSymbol` both emit identifier offsets): `SymbolInformation` has no `selectionRange`, so for workspace-symbol hits this collapses into re-fetching `documentSymbols` and matching — relocating the same brittleness rather than removing it.
- *C / D — resolve-layer tree-sitter fallback + observability only (point fix):* would have closed the disambiguation branch but left the `CodeSymbol` conflation un-named and reintroduceable by any future code path that bypasses refine. Rejected because the repo's evidence-strictness philosophy prefers honest, compiler-enforced distinctions over silent fall-back.
- *Additive migration with deprecated `line`/`character` aliases:* rejected because aliases re-introduce the exact silent conflation that *is* the bug.

**Consequences.** Big-bang breaking change, sanctioned by the repo's pre-release policy ("intentional breaking changes to package APIs … are allowed"). ~8–10 test files across `supi-code-intelligence` and `supi-lsp` must be rewritten: they currently encode the bug past CI by mocking the provider and asserting `displayCharacter` values they fed in. The rewrite must encode the new invariant (disambiguation candidates resolve to the *name* anchor, not the declaration) and add at least one real-LSP integration test for resolve — the kind that would have caught this class. `tg-*`/`sp-*` identity semantics change (position removed from the hash); within-session handles remain stable across re-resolve (cross-session persistence is already out of scope). Loud `unavailable` for `calleesAt`/`rename` may *reduce apparent coverage* for files where `nameAnchor` cannot be derived, but this is honest-unavailable rather than silent-wrong, consistent with the stack's stated behavior.

**Note.** This ADR occupies `0003`, not `0002`, because `packages/supi-code-intelligence/README.md` references a `docs/adr/0002-refactor-planner-applier-split.md` that did not exist on disk at the time this ADR was written. The planner/applier invariant is documented in prose in `packages/supi-code-intelligence/CLAUDE.md` and now also in `docs/adr/0002-refactor-planner-applier-split.md`, which resolves the dangling README reference. `code_impact` intentionally has no declaration-anchor guard: it runs on `references`, a position-tolerant consumer (see Decision), so a declaration anchor still yields correct results where `calleesAt`/`rename` would not.
