# Evidence lists disclose truncation for public tool evidence

**Context.** `supi-code-intelligence` public tools cap many result sections for readability, but capped markdown and structured `details` can drift or silently imply completeness. Silent truncation violates the project’s honest-correctness standard just as much as silent fallback or anchor conflation: a user or agent must be able to distinguish “there are no more results” from “more evidence exists but was not shown.”

**Decision.** Public tool evidence and generated executable/check command lists are represented as bounded evidence/actionable lists with shared completeness metadata. Normal public-tool paths compute exact totals; `maxResults` is a display cap, not a collection cap. Each capped list discloses truncation in markdown using the standard local note `_(showing X of Y; N omitted)_`, and structured tool details keep the existing aggregate `omittedCount` while adding per-list metadata keyed by stable dotted identifiers such as `references.locations`, `tests.files`, `tests.labels`, `health.diagnostics`, or `refactor.edits`. Evidence-list counts use evidence atoms, not rendered grouping rows: reference locations count as references even when grouped by file, individual diagnostics count as diagnostics even when grouped by file, and test files and test labels are separate lists. Unknown totals are allowed only for exceptional interruption or provider-limited results — timeout, safety-limit, interrupted enumeration, or an upstream provider that cannot expose a true total — and must carry an explicit partial reason instead of pretending exact completeness. The Evidence list abstraction remains internal to `supi-code-intelligence`; it is not promoted to `supi-code-runtime` until another package has a concrete need for it.

**Considered Options (rejected).** Renderer-only notes were rejected because markdown and structured details would drift. Details-only metadata was rejected because human-readable output could still look complete. Per-tool custom truncation logic was rejected because it would encode the same invariant differently across tools. Treating `maxResults` as a collection cap was rejected because it prevents exact omission counts on normal paths. Promoting Evidence list to `supi-code-runtime` was rejected as premature generalization.

**Consequences.** The migration is a public-surface big bang across all public tool evidence, including orientation briefs, graph relations, search results, resolve candidates, health diagnostics/status evidence, inspect/code-action facts, impact evidence, refactor edit previews, and generated verification commands. Prose guidance lists such as `nextQueries` remain guidance chrome and are out of scope. Tests must assert both markdown disclosure and matching structured metadata for capped evidence/actionable lists. Grouped reference rendering changes from capping files to capping reference locations. Path rendering bugs in public evidence, such as `code_health` displaying `CONTEXT.md` as `ONTEXT.md`, are in scope for this correctness batch.
