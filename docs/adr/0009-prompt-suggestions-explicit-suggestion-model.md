# Prompt suggestions require an explicit suggestion model

`@mrclrchtr/supi-prompt-suggestions` will use a single `model` setting whose value is either `disabled` or a canonical `provider/model-id` suggestion model. Model selection will mirror `@mrclrchtr/supi-review`: selectable models come from PI's scoped `enabledModels` set, the current session model is preferred only when it is inside that scoped set, and arbitrary unscoped model ids are not offered. Unlike `supi-review`, prompt suggestions run automatically after assistant responses, so the chosen suggestion model is persisted in `promptSuggestions.model` rather than asked for every run. The setting uses SuPi's normal scoped config behavior: global default plus optional project override via `/supi-settings`. The settings UI exposes it as a picker rather than freeform text: `disabled` first, then selectable models from the scoped model set. SuPi's settings registry will pass the active `ExtensionContext` to settings callbacks so this picker can use PI's `modelRegistry` and current session model without duplicating or weakening `supi-review` model-selection semantics. The scoped-model selection helpers will be extracted from `supi-review` into a shared `@mrclrchtr/supi-core` API callable by both packages. If the persisted model later falls out of `enabledModels` or lacks configured auth, generation treats prompt suggestions as disabled rather than silently widening to another model. The `provider/model-id` value is parsed at the first slash because PI provider ids do not contain slashes while many model ids do. The default is `disabled`, and the package will not implicitly call the current PI model. This sacrifices out-of-the-box suggestions, but avoids surprising background model calls and lets users choose the cheap, fast model intended for advisory prompt suggestions.
